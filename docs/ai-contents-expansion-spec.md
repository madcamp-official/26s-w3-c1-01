# AI 콘텐츠 확장 퀴즈 — 기능 명세서

> 실존 사건과 AI가 창작한 "그럴듯한 가상 사건"을 같은 톤으로 서술한 지문을 제시하고,
> 사용자가 실화/AI창작 여부를 맞히는 퀴즈형 콘텐츠 서비스

## 1. 개요

### 1.1 목적
- 짧은 사건 서술(지문)을 보여주고 사용자가 "실화"인지 "AI 창작"인지 맞히는 퀴즈 경험을 제공한다.
- 정답 공개 시 짧은 코멘트로 재미와 학습 효과(진짜였다면 어떤 사건인지, 왜 그럴듯했는지)를 더한다.

### 1.2 예시 (기준 샘플)
```
지문: 1932년, 한 나라의 군대가 실제로 에뮤(타조 닮은 새)와의 전쟁을 벌였다.
      농작물을 망치는 에뮤 떼를 막기 위해 기관총까지 동원한 정식 군사 작전을
      펼쳤지만, 에뮤들이 흩어지고 빠르게 도망다니는 바람에 결국 군대가 패배를
      인정하고 철수했다.

선택지: 실화 / AI 창작
정답: ✅ 실화 (1932년 호주의 "에뮤 전쟁")
공개 후 코멘트: "네, 진짜입니다. 심지어 의회에서까지 논의된 정식 군사작전이었어요."
```
이 구조(지문 → 사용자 선택 → 정답/근거 공개 → 코멘트)를 모든 콘텐츠 카드의 기준 포맷으로 삼는다.

### 1.3 역할 구분 기준
본 명세서는 **프론트엔드(FE) / 백엔드(BE) / AI 파이프라인** 세 축으로 기능을 나눈다.

| 역할 | 책임 범위 |
|---|---|
| 프론트엔드 | 지문 노출, 사용자 응답 수집, 정답/코멘트 공개 연출, 점수·진행 상태 표시 |
| 백엔드 | 콘텐츠 카드 저장/서빙, 퀴즈 세션·정답 판정·점수 로직, AI 파이프라인 트리거 및 결과 적재 |
| AI 파이프라인 | 실화 지문 생성(사실 기반 각색), AI창작 지문 생성(그럴듯한 가상 사건), 정답 근거·코멘트 생성, 품질 검증 |

---

## 2. 용어 정의

| 용어 | 정의 |
|---|---|
| 콘텐츠 카드 (Content Card) | 하나의 퀴즈 단위. 지문 1개 + 정답(실화/AI창작) + 근거 + 공개 코멘트로 구성 |
| 지문 (Body) | 사용자에게 우선 노출되는 사건 서술 텍스트 |
| 실화 카드 | 실제 존재했던 사건을 각색해 서술한 카드 (`type = REAL`) |
| AI창작 카드 | 실화처럼 보이도록 AI가 만들어낸 가상 사건 카드 (`type = FABRICATED`) |
| 정답 근거 (Answer Fact) | 실화 카드의 경우 실제 사건명·연도·출처, AI창작 카드의 경우 "왜 가짜인지/무엇을 모티프로 했는지" |
| 공개 코멘트 (Reveal Comment) | 정답 공개 시 노출되는 짧은 부연 설명 |

---

## 3. 사용자 플로우

1. 사용자가 카드 목록/시작 화면에서 퀴즈를 시작한다.
2. 서버가 다음 콘텐츠 카드의 **지문만** 전달한다 (정답·근거·코멘트는 미포함).
3. 사용자가 "실화" 또는 "AI 창작" 중 하나를 선택한다.
4. 클라이언트가 응답을 서버에 제출한다.
5. 서버가 정답 판정 후 `정답 여부 + 정답 근거 + 공개 코멘트`를 반환한다.
6. 클라이언트가 정답 공개 UI(정오답 표시, 근거, 코멘트)를 렌더링하고 점수를 갱신한다.
7. 다음 카드로 이동하거나 결과 요약 화면으로 이동한다.

## 4. 데이터 모델

### 4.1 ContentCard (콘텐츠 카드)

| 필드 | 타입 | 설명 |
|---|---|---|
| `id` | string | 카드 고유 ID |
| `category` | string | 분류 태그 (예: 역사, 과학, 사회, 전쟁 등) |
| `body` | text | 지문 텍스트 |
| `type` | enum(`REAL`, `FABRICATED`) | 실화 / AI창작 여부 (정답) |
| `answer_title` | string | 정답 요약 (예: "1932년 호주의 에뮤 전쟁") |
| `answer_fact` | text | 정답 근거 상세 (실화: 사건 개요·출처 / AI창작: 무엇을 모티프로 창작했는지) |
| `reveal_comment` | text | 정답 공개 시 노출 코멘트 |
| `source_url` | string? | 실화 카드의 출처 링크 (검증용, nullable) |
| `difficulty` | enum(`EASY`,`MEDIUM`,`HARD`)? | 난이도 (선택) |
| `status` | enum(`DRAFT`,`VERIFIED`,`PUBLISHED`,`REJECTED`) | AI 파이프라인 검수 상태 |
| `created_at` / `updated_at` | datetime | 생성/수정 시각 |

### 4.2 QuizSession / Answer (선택적, 점수 기능용)

| 필드 | 타입 | 설명 |
|---|---|---|
| `session_id` | string | 사용자 퀴즈 세션 ID |
| `card_id` | string | 응답 대상 카드 |
| `user_choice` | enum(`REAL`,`FABRICATED`) | 사용자 선택 |
| `is_correct` | boolean | 정답 여부 |
| `answered_at` | datetime | 응답 시각 |

## 5. 역할별 기능 명세

### 5.1 프론트엔드 (FE)

| 기능 ID | 기능명 | 상세 설명 |
|---|---|---|
| FE-01 | 카드 지문 노출 | `body`만 표시, 정답 관련 필드는 응답 전까지 절대 노출 금지 |
| FE-02 | 선택지 UI | "실화 / AI 창작" 2지선다 버튼 제공, 선택 전 재선택 가능 |
| FE-03 | 응답 제출 | 선택 즉시 BE에 `card_id + user_choice` 전송 |
| FE-04 | 정답 공개 연출 | 정오답 여부(✅/❌), `answer_title`, `answer_fact`, `reveal_comment` 순차 노출 |
| FE-05 | 점수/진행 표시 | 현재까지 정답 수, 진행률(예: 3/10) 표시 |
| FE-06 | 다음 카드 이동 | 정답 공개 후 "다음" 액션으로 다음 카드 요청 |
| FE-07 | 결과 요약 화면 | 세트 종료 시 총점, 맞은/틀린 카드 리스트, 카드별 재열람 제공 |
| FE-08 | 카테고리/난이도 필터 | 시작 화면에서 카테고리·난이도 선택해 퀴즈 세트 구성 (선택 기능) |
| FE-09 | 오답 노출 방지 가드 | API 응답에 정답 필드가 섞여 와도 화면에 절대 선반영되지 않도록 상태 분리 (지문 로드 상태 ≠ 정답 공개 상태) |

### 5.2 백엔드 (BE)

| 기능 ID | 기능명 | 상세 설명 |
|---|---|---|
| BE-01 | 카드 목록/단건 조회 API | `GET /cards` (필터: category, difficulty), `GET /cards/{id}` — **정답 필드 제외**하고 반환 |
| BE-02 | 퀴즈 세트 생성 API | `POST /quiz-sessions` — 카테고리/난이도/개수 조건으로 카드 N개를 셔플해 세션 생성 |
| BE-03 | 응답 제출 및 채점 API | `POST /quiz-sessions/{id}/answers` — `card_id + user_choice` 수신, `type`과 비교해 채점 후 정답 필드 포함 응답 반환 |
| BE-04 | 세션 결과 집계 API | `GET /quiz-sessions/{id}/result` — 총점, 카드별 정오답 리스트 반환 |
| BE-05 | 콘텐츠 카드 저장소 | ContentCard CRUD, `status` 기반 노출 제어 (`PUBLISHED`만 사용자 API에 노출) |
| BE-06 | AI 파이프라인 트리거 | 신규 카드 생성 요청을 AI 파이프라인에 위임하고 결과를 `DRAFT` 상태로 적재 |
| BE-07 | 검수 워크플로 | `DRAFT → VERIFIED/REJECTED → PUBLISHED` 상태 전이 관리 (수동 검수 또는 AI 자동검증 결과 반영) |
| BE-08 | 정답 유출 방지 | 카드 조회 API 응답 스키마에서 `type`, `answer_title`, `answer_fact`, `reveal_comment`를 물리적으로 분리 직렬화 (응답 전 노출 금지) |
| BE-09 | 중복/재출제 방지 | 동일 세션 내 카드 중복 출제 방지, (선택) 사용자별 최근 풀이 이력 기반 재출제 제한 |

### 5.3 AI 파이프라인

| 기능 ID | 기능명 | 상세 설명 |
|---|---|---|
| AI-01 | 실화 소재 수집 | 실제 사건 소재(주제/연도/출처)를 수집·선정 (검색/지식베이스 기반) |
| AI-02 | 실화 지문 생성 | 수집된 사실을 바탕으로 `body` 각색 생성 (허구 첨가 없이 사실 유지, 흥미 위주 문체로 재구성) |
| AI-03 | AI창작 지문 생성 | 실화 카드와 동일한 톤·분량·문체로 "있을 법한" 가상 사건 생성 (실존 사건과 혼동되지 않도록 실제 고유명사·연도 재사용 금지) |
| AI-04 | 정답 근거 생성 | 실화: 사건 요약·출처(`answer_fact`, `source_url`) 생성 / AI창작: 창작 모티프·왜 그럴듯한지 설명 생성 |
| AI-05 | 공개 코멘트 생성 | 정답 공개 시 노출할 짧고 흥미로운 코멘트(`reveal_comment`) 생성 |
| AI-06 | 사실 검증 (실화 카드) | 생성된 실화 지문의 사실 정합성 검증 (연도·고유명사·사건 전개 오류 탐지), 실패 시 `REJECTED` |
| AI-07 | 개연성 검증 (AI창작 카드) | 가상 사건이 시대적 모순·논리적 비약 없이 "그럴듯한지" 검증, 지나치게 허무맹랑하거나 반대로 실화와 구분 불가할 만큼 유사한 경우 재생성 |
| AI-08 | 난이도 태깅 | 생성된 카드의 변별력(실화/창작 구분 난이도) 추정 후 `difficulty` 부여 |
| AI-09 | 결과 적재 | 생성·검증 완료된 카드를 BE에 `DRAFT`/`VERIFIED` 상태로 전달 |

## 6. API 요약 (BE ↔ FE)

| Method | Endpoint | 설명 | 정답 필드 포함 여부 |
|---|---|---|---|
| POST | `/quiz-sessions` | 퀴즈 세트 생성 | ❌ |
| GET | `/quiz-sessions/{id}/next-card` | 다음 카드(지문만) 조회 | ❌ |
| POST | `/quiz-sessions/{id}/answers` | 응답 제출 및 채점 | ✅ (응답에만 포함) |
| GET | `/quiz-sessions/{id}/result` | 세트 결과 요약 | ✅ |
| (관리) POST | `/cards/generate` | AI 파이프라인 카드 생성 트리거 | - |
| (관리) PATCH | `/cards/{id}/status` | 카드 검수 상태 변경 | - |

## 7. 비기능 요구사항

- **정답 유출 방지**: 지문 조회 시점과 정답 공개 시점의 API 응답 스키마를 반드시 분리한다. (FE-09, BE-08)
- **콘텐츠 신뢰성**: 실화 카드는 반드시 AI-06 사실 검증을 통과해야 `PUBLISHED` 가능.
- **변별력 확보**: AI창작 카드가 지나치게 허술하거나(AI-07 실패) 반대로 실화보다 더 그럴듯해서 검증 불가능한 수준이 되지 않도록 개연성 검증 기준을 둔다.
- **캐싱**: 동일 카드 지문은 AI 재생성 없이 캐시된 결과를 재사용한다 (BE-05).
