# 내가 안다고 착각한 단어 — 작업 노트

몰입캠프 3주차 공통과제 III · **옵션 2 (Launch & Spread)**

이 문서는 README가 아니라 **작업/핸드오프 노트**다. 구조, 결정의 이유, 백엔드
연결 지점, 배포 전에 반드시 해야 할 것을 담는다. 다른 사람이나 다른 환경의

## 1. 이 제품이 노리는 것

**"어 나 이거 알지! ...막상 쓰려니 힘드네?"라는 5초.**

그게 전부다. 나머지 화면은 그 5초로 데려가거나, 그 5초를 확인시켜 줄 뿐이다.
학습 플랫폼이 아니고 채점 서비스도 아니다. 30초짜리 경험이고, 그게 공유되는지
확인하는 실험이다.

흐름:

```
랜딩(= 1번 단어의 질문) → 안다  → 설명 쓰기(5초 게이트) → 정답 공개 + 자기판정 → 다음
                          모른다 → 정답 공개만                                   → 다음
                                                              ×5 → 착각 지수 + 공유
```

**랜딩이 곧 첫 단어의 질문 화면이다.** "시작" 버튼을 누르고 똑같은 질문을 또 받으면
루프에 빠진 것처럼 보여서, 랜딩에서 바로 안다/모른다를 답하고 넘어간다.
그 답은 `lib/handoff.ts`가 첫 단어로 넘긴다.

## 2. 제1원칙: 읽어야 하는 글자를 최소화한다

**이게 다른 모든 결정에 우선한다.** 한때 결과 화면에 정의 + 핵심요소 + 해설 문단
+ 통계 문장까지 있었고, 퀴즈 하나 풀고 논문을 읽는 꼴이었다.

지금 화면에 남은 산문:
- 랜딩: "설명할 수 있나요?" + 훅 한 줄
- 입력: "설명해 보세요" (placeholder) + "잠깐만 더 생각해 보세요"
- 공개: 정답 한 문장 + 짧은 칩 3개 + "내 설명, 맞았나요?"
- 요약: 한 줄짜리 verdict

**설명·해설·기능을 추가하기 전에 이 원칙을 먼저 확인할 것.**

## 3. 핵심 설계 결정과 그 이유

### 3.1 왜 LLM 채점을 안 쓰는가

**판정이 필요 없기 때문이다.** "막상 쓰려니 힘드네"는 텍스트 박스 안에서 이미
일어난다. 채점 이전에, 누가 틀렸다고 말해주기 전에 스스로 안다.

이건 실제로 연구된 현상이고(illusion of explanatory depth, Rozenblit & Keil),
원래 실험 방법이 ①얼마나 안다고 생각하는지 → ②설명 써보기 → ③다시 평가하기다.
**점수가 떨어지는 것 자체가 발견**이고 연구자가 답을 채점하지 않는다.

LLM을 붙였을 때의 실제 비용은 코드가 아니라 그 바깥에 있다:
- **지연 2~5초** — 감정의 정점에서 로딩 스피너가 뜬다. 가장 아픈 지점
- API 키와 결제 수단이 필요하고, 공개된 유료 엔드포인트라 어뷰징에 노출된다
- 전부 정적(SSG)이던 배포에 서버 함수가 생긴다
- 한국어 설명을 가끔 야박하게 판정한다 → 사용자는 억울해한다

**붙인다면 실제 사용자 설명이 쌓인 뒤에.** `SessionAnswer.text`에 원문이 남으므로
그게 판단 근거가 된다. `lib/stats.ts`처럼 seam이 있어 교체는 쉽다.

### 3.2 키워드 매칭은 힌트지 판정이 아니다

**이 구분이 전부다.**

키워드 매칭으로 맞다/틀리다를 **단정하면** 함정이다. "국민이 주인인 나라"라고 잘
썼는데 '주권'이 없다고 틀렸다고 하면, **사용자는 자기가 틀렸다고 느끼는 게 아니라
앱이 고장났다고 느낀다.** 한국어는 조사·어미 변형까지 겹쳐서 더 심하다.
**부정확한 판정은 판정이 없는 것보다 나쁘다.**

그래서 `RevealPanel`의 초록 체크는 **참고용**이다. 체크가 0개여도 "맞았다"를 누를
수 있고, 3개여도 "틀렸다"를 누를 수 있다. 최종 판정은 언제나 사용자가 한다.
매칭은 "아 나 이 부분을 안 썼구나"를 눈에 보이게 해줄 뿐이다.

오탐을 줄이려고 `Keyword.match`에 표현 변형을 여러 개 받는다
(`주권` ← `["주권", "주인", "권력은 국민"]`). 반대로 너무 넓은 말("법", "것")은
넣지 말 것 — 아무 답에나 체크가 켜진다.

### 3.3 자기판정이 무르지 않은 이유

내가 쓴 "다수결로 정하는 거"와 정답 "국민주권 + 법치 + 기본권"이 **나란히 놓이면
부정할 수가 없다.** 그리고 거짓말해서 🟩 받아봐야 결과 격자가 심심해질 뿐이라
속일 동기도 없다.

`keywords`가 판정의 실질적 기준이다. 문단 하나를 던져주면 "대충 맞는 것 같은데?"로
넘어가지만, "법치 ○ — 이거 안 썼네"는 부정할 수가 없다. **그래서 3개로 유지한다.**
늘리면 읽을거리가 되고 판정이 무뎌진다.

판정은 3단계다: **맞았다 / 애매하다 / 틀렸다.** 2단계면 애매한 답을 어디에 넣을지
몰라서 아무 데나 찍게 되고, 그러면 착각 지수가 의미를 잃는다.

### 3.4 5초 게이트

`ExplainBox.tsx`. 제출 버튼이 5초간 잠기고 카운트다운(5→4→3→2→1)이 돈다.

**안 막으면 빈칸으로 즉시 넘겨버린다.** 그러면 착각이 드러나기 전에 화면이
지나가고 아무 일도 일어나지 않는다. 강제로 빈 텍스트 박스를 마주하게 하는 5초가
곧 제품이다.

남은 시간은 **마감 시각에서 역산**한다. `setInterval` 호출 횟수를 세면 탭이
백그라운드로 갔을 때 5초보다 오래 걸린다.

**빈칸 제출은 막지 않는다.** 안다고 해놓고 한 글자도 못 쓰는 것 — 그게 가장 순수한
착각이라 오히려 살려둬야 한다. 별도의 "포기" 버튼이 필요 없는 이유다.
(이벤트에 `gaveUp: true`로 기록된다.)

**Enter로 제출된다.** 단, `e.nativeEvent.isComposing`을 반드시 확인할 것 —
한글 조합 중의 Enter는 글자를 확정하는 키라서, 가로채면 "민주주의"를 치다가
마지막 글자를 확정하는 순간 제출돼 버린다. 줄바꿈은 Shift+Enter.

### 3.5 "모른다"는 정직한 퇴로다

모른다를 누르면 설명을 쓰지 않고 정답만 보고 넘어간다. **모른다고 인정한 사람은
이 앱의 관심사가 아니다** — 착각은 "안다"고 한 사람에게만 일어난다.

대가로 결과 격자가 ⬜로 심심해진다. 전부 모른다를 누르면 ⬜⬜⬜⬜⬜가 나와서
공유할 게 없어지므로, 게으른 우회로가 스스로를 처벌한다.

### 3.6 착각 지수와 이모지 격자

Wordle이 퍼진 건 게임이 아니라 **정답을 노출하지 않는 이모지 격자** 때문이었다.
단어 하나 풀고 "나 민주주의 틀림"은 공유할 이유가 없어서, 5단어를 한 세션으로
묶고 요약 화면을 만들었다.

| 이모지 | 조건 | 의미 |
| --- | --- | --- |
| 🟥 | 안다 + "틀렸다" | **착각** ← 이 제품의 주제 |
| 🟨 | 안다 + "애매하다" | 반쯤 알았음 |
| 🟩 | 안다 + "맞았다" | 진짜 알았음 |
| ⬜ | 모른다 | 판정 없음 |

**착각 지수 = 🟥의 개수.** 🟨는 세지 않는다 — 착각은 '확실히 틀림'으로만 인정한다.
공유 텍스트에는 단어명도 정답도 일절 들어가지 않는다. 스포일러가 있으면 받은
사람이 풀 이유가 사라지고 확산이 멈춘다.

"모른다"에 ⬜를 주는 이유: 판정 자체가 없기 때문이다. 객관식 시절의
"찍어서 맞음"(🟦)은 주관식에서 존재하지 않는다.

### 3.7 세션 = sessionStorage

로그인이 없으므로 `sessionId`가 유일한 사용자 식별자다. 탭을 닫으면 사라지게
해서 "한 번의 방문 = 한 세션"으로 맞췄다.

공유 링크로 `/w/inertia`에 바로 들어오면 **그 단어가 세션의 1번**이 되고 나머지가
뒤에 붙는다. (`lib/session.ts`의 `ensureSession`)

라우트를 단어별로 쪼갠 이유는 **공유 링크가 특정 단어를 가리켜야 해서**다.
한 단어 안의 단계는 라우트 이동 없이 클라이언트 상태로 넘긴다 — 제출 후
로딩이 걸리면 착각이 드러나는 순간의 긴장이 풀린다.

### 왜 백엔드를 Next.js API Routes가 아니라 별도 서비스로 뒀나

Caddy가 `/api/*`를 백엔드로, 나머지를 Next.js로 넘긴다. 브라우저 입장에서는
**동일 출처**라 CORS도 혼합 콘텐츠 차단도 발생하지 않는다 — API Routes를 썼을
때와 똑같다. 대신 프론트를 나중에 Vercel로 옮겨도 백엔드는 그대로 살아 있고,
백엔드를 팀원이 독립적으로 맡을 수 있다. 프론트는 여전히 전부 정적(SSG/ISR)이다.

### 왜 `useSyncExternalStore`인가

sessionStorage는 외부 저장소라 렌더 중에 읽으면 SSR 하이드레이션이 깨진다.
effect에서 `setState`로 끌어오는 방식은 React 19 린트 룰
(`react-hooks/set-state-in-effect`)에 걸리고 캐스케이딩 렌더를 만든다.
그래서 구독 방식으로 갔다. **스냅샷은 참조가 안정적이어야 하므로**
`lib/session.ts`에서 원본 문자열이 같으면 같은 객체를 돌려주도록 캐싱한다.
이 캐시를 빼면 무한 루프가 난다.

## 6. 백엔드

### 6.0 한눈에

```
브라우저 ──https──> Caddy ──/api/*──> api (Fastify) ──> Postgres
                      └────그 외────> web (Next.js)
```

API는 라우트 4개가 전부다:

| 메서드 | 경로 | 하는 일 |
| --- | --- | --- |
| `POST` | `/api/events` | 이벤트 수집. **무슨 일이 있어도 204** |
| `GET` | `/api/stats/:wordId` | `{confidentCorrectRate}` 또는 `null`(표본 부족) |
| `GET` | `/api/admin/summary` | 퍼널·유입별·단어별·추천 목록. `Bearer $ADMIN_TOKEN` |
| `GET` | `/api/health` | DB까지 찔러 본다 |

수치 보기:
```bash
curl -s -H "Authorization: Bearer $ADMIN_TOKEN" https://<도메인>/api/admin/summary | jq
```

### 6.1 이벤트 수집 — `lib/analytics.ts` → `server/src/events.ts`

`track()`이 4개 싱크(console / localStorage / GA / `/api/events`)로 흘린다.
전송은 `sendBeacon`을 먼저 쓰고 실패 시 `fetch(keepalive)`로 폴백한다.

⚠️ **저장 여부를 정하는 건 프론트가 아니라 서버의 화이트리스트다**
(`server/src/events.ts`의 `ALLOWED_PROPS`). 표에 없는 필드는 조용히 버려진다.
프론트에 새 prop을 추가하고 서버 표를 안 고치면 **에러 없이 빈칸이 된다.**

이벤트 페이로드 스키마:

```ts
{
  event: EventName,        // 아래 10종
  sessionId?: string,      // 익명 UUID (sessionStorage)
  wordId?: string,
  timestamp: number,
  source?, medium?, campaign?,  // UTM (세션 내내 유지)
  ...이벤트별 추가 필드
}
```

| 이벤트 | 발생 시점 | 추가 필드 |
| --- | --- | --- |
| `landing_view` | 랜딩 진입 | — |
| `test_start` | 랜딩에서 답함 / 공유링크 직접 진입 | `wordId`, `entry: "landing" \| "direct"` |
| `confidence_selected` | 안다/모른다 선택 | `wordId`, `knew: boolean` |
| `answer_submitted` | 설명 제출 (안다 경로만) | `wordId`, `knew`, `answerLength`, `gaveUp` |
| `result_view` | 정답 공개 | `wordId`, `knew` |
| `self_judged` | 자기판정 (안다 경로만) | `wordId`, `judgment`, `gaveUp` |
| `next_word_click` | 다음 단어로 | `wordId`, `nextWordId?` |
| `retry_click` | 다시 하기 | `from?` |
| `share_click` | 공유 버튼 | `kind: "result" \| "word"`, `wordId?` |
| `session_complete` | 요약 화면 도달 | `total`, `claimed`, `illusions`, `durationMs` |
| `word_suggested` | 단어 추천 제출 | `word`, `length` — `suggestions` 테이블로도 들어감 |

`test_start`와 `session_complete`는 **세션당 정확히 한 번**만 발생한다.
이 둘의 비율이 곧 완주율(퍼널의 핵심 지표)이다.

`gaveUp`(안다고 해놓고 한 글자도 못 씀)이 **이 제품 고유의 지표**다. 이게 높으면
착각이 제대로 작동하고 있다는 뜻이다.

### 6.2 사용자가 쓴 설명 원문 — 저장하지 않기로 했다 ★

`SessionAnswer.text`는 브라우저 sessionStorage에만 있고 **서버로 가지 않는다.
의도적인 결정이다** (2026-07-17).

- **왜 안 받나**: 자유 입력이라 개인정보가 섞일 수 있다. 안 받으면 유출될 것도,
  지켜야 할 것도, 파기 정책을 세울 것도 없다. 1주일짜리 실험이 감당할 수 있는
  범위를 넘는다.
- **왜 괜찮나**: 우리가 알아야 하는 착각률은 `confidence_selected` + `self_judged`
  만으로 전부 나온다. 원문은 "있으면 좋은" 데이터지 지표에 필요한 데이터가 아니다.
- **대가**: 사람들이 민주주의를 실제로 어떻게 설명하는지 영영 못 본다.
  `keywords` 오탐 확인도 서버 데이터로는 불가능하다(§5 참고). LLM 채점을 붙일
  가치가 있는지도 데이터로는 답할 수 없다. **알고 버린 것이다.**
- **뒤집으려면**: `server/src/events.ts`의 `ALLOWED_PROPS`에 추가해야만 저장된다.
  그 표를 고치는 게 이 결정을 뒤집는 행위다. 팀 합의 없이 하지 말 것.

### 6.3 통계 조회 — `lib/stats.ts` → `GET /api/stats/:wordId`

응답:
```ts
{ confidentCorrectRate: number } | null
```
`confidentCorrectRate` = (해당 단어에 knew이고 judgment=correct인 세션 수) / (knew인 세션 수)

⚠️ **`null`은 에러가 아니다.** 표본(knew라고 답한 세션)이 `MIN_SAMPLE`(30) 미만이면
서버가 일부러 null을 준다. UI는 null이면 문구를 통째로 숨긴다. 자세한 이유는
`server/src/stats.ts` 주석.

호출 경로가 둘이라는 점만 주의:
- 랜딩(`app/page.tsx`) — **서버**에서. 상대 경로를 못 쓰므로 `API_INTERNAL_URL` 사용
- 정답 공개(`WordPlay`) — **브라우저**에서. 상대 경로 `/api`

도커 빌드 중에는 API가 없어서 항상 null이 나오고, 그래서 `getWordStats()`는
절대 throw하지 않는다. 통계 하나 때문에 빌드가 깨지거나 화면이 안 뜨면 안 된다.

## 7. 단어 추가하기

`data/words.ts`의 `WORDS` 배열에 추가하면 라우트·SSG·세션 순서에 자동 반영된다.
다른 파일은 손댈 필요 없다. 필드는 4개뿐이다:

```ts
{
  id: "algorithm",          // URL 슬러그. 바꾸면 기존 공유 링크가 깨진다
  word: "알고리즘",
  definition: "문제를 풀기 위한, 유한한 단계로 이루어진 명확한 절차. ...",
  keywords: [               // 3개 유지. match에 표현 변형을 넉넉히
    { label: "절차·단계", match: ["절차", "단계", "순서", "과정"] },
    { label: "문제 해결", match: ["문제", "해결", "풀"] },
    { label: "유한함",   match: ["유한", "끝", "종료", "정해진"] },
  ],
}
```

**단어 선정 기준이 제품의 전부다.** 지켜야 할 조건:

1. **누구나 안다고 답할 만큼 익숙할 것.** 아무도 모르는 단어는 전원이 "모른다"를
   눌러서 ⬜만 쌓이고 착각이 발생하지 않는다.
2. **막상 설명하면 막힐 것.** 뻔한 단어(사과, 의자)는 그냥 설명된다.
3. 좋은 후보는 **흔한 오해가 딱 하나 박혀 있는 단어**다. 관성(힘이라고 믿음),
   진화(진보라고 믿음), 인플레이션(상태라고 믿음)처럼.

AI로 대량 생성할 때는 "어려운 단어"를 요구하면 안 된다. **"익숙한데 설명하면
막히는 단어"**를 요구해야 한다. 생성 후 `definition` 사실 검증은 사람이 한다.

## 9. 진행 기록

### 2026-07-17 — 초기 MVP (객관식)

**처음 계획**: 자기평가 → 주관식 설명 → 정답 공개 → 통계.

**실제로 만든 것**: 주관식은 LLM 채점이 필요하다고 판단해 객관식으로 내렸다.
대신 오답을 '흔한 오해'로 채우고 점수를 착각률로 바꿔 주제를 유지하려 했다.

**계획에서 바뀐 것**: 단어 1개 단위 → 5개 세션 + 요약 화면(공유 엔진).
통계는 가짜임을 UI에 강제 노출하는 플래그를 넣었다.

### 2026-07-17 — 1차 개선: 텍스트 대량 삭제

**문제**: "내가 사용자였으면 바로 껐을 것 같다." 원인은 읽어야 하는 글자가 너무 많았다.

**들어낸 것**: 결과 화면의 해설 패널 전체(정의·핵심요소·해설 문단·오해 설명·통계
문장), 4단계 자기평가와 부연설명, "정답 확인" 버튼. `Word` 타입이 8필드 → 4필드.

**발견**: 해설을 지웠는데 정보가 부족해지지 않았다. **정답 보기 자체가 정의라서**
초록색으로 켜는 것만으로 설명이 됐다. 해설 문단은 처음부터 같은 말의 반복이었다.

### 2026-07-17 — 2차 개선: 객관식 → 주관식 (현재)

**문제**: 객관식은 재인(recognition)이라 소거법으로 맞히고 "난 역시 알았네"로
끝난다. **착각이 드러나기는커녕 강화된다.** 처음부터 알고 있던 문제였는데
LLM 채점을 피하려다 객관식을 택했던 것이다.

**돌파구**: **판정이 필요 없다는 것.** "막상 쓰려니 힘드네"는 텍스트 박스 안에서
채점 이전에 이미 일어난다. LLM 없이도 원하는 5초를 그대로 얻을 수 있었다.

**바꾼 것**:
- 보기 4개 삭제 → 자유 입력(`ExplainBox`). `Word`에서 `choices` 제거,
  `definition`/`keyPoints` 부활 (이번엔 읽을거리가 아니라 **자기판정 기준**으로)
- **5초 게이트** — 제출 버튼이 5초간 잠긴다. 안 막으면 빈칸으로 즉시 넘겨서
  아무 일도 일어나지 않는다
- 채점 = 본인. 내 설명과 정답을 나란히 놓고 [맞았다]/[틀렸다] 1탭
- 격자 4상태 → 3상태 (모른다는 판정이 없으므로 🟦이 사라짐)
- 사용자가 쓴 원문을 세션에 보관 → LLM 필요성 판단의 근거로

**검토했다 버린 것**: 키워드 매칭. "국민이 주인인 나라"를 틀렸다고 하는 순간
사용자는 앱이 고장났다고 느낀다. **부정확한 판정은 판정 없음보다 나쁘다.**

**검증**: 5초 게이트가 정확히 5초(5→4→3→2→1) 잠기는 것, 안다/모른다/빈칸 제출
세 경로, 격자와 착각 지수 일치, 이벤트 10종, `gaveUp` 기록. 빌드/린트 통과.

### 2026-07-17 — 3차 개선: 사용성 다듬기

첫 사용 후 나온 피드백 7건을 반영했다. 방향은 유지, 마찰만 제거.

1. **랜딩과 첫 단어가 같은 질문을 두 번 함** → 랜딩의 "시작" 버튼을 안다/모른다로
   바꾸고, 그 답을 `lib/handoff.ts`로 첫 단어에 넘겨 질문 화면을 건너뛴다.
   화면이 하나 줄었다.
2. **Enter로 제출.** 한글 조합 중 Enter를 걸러내는 게 핵심(`isComposing`).
   안 그러면 마지막 글자를 확정하는 순간 제출된다.
3. **`keyPoints` 문장 → `keywords` 칩 + 초록 체크.** 답에 키워드가 있으면 켜진다.
   **채점이 아니라 힌트다** (3.2 참고). 오탐을 줄이려고 동의어를 여러 개 받는다.
4. **판정에 "애매하다" 추가** → 격자가 4상태(🟩🟨🟥⬜)로. 착각 지수는 🟥만 센다.
5. 결과 화면에 **내가 쓴 설명 전체**를 보여준다. 남의 글이 아니라 자기 글이라
   읽는 부담이 아니라 보상이 된다.
6. 모든 화면이 375×812에서 스크롤 없이 들어가는 것을 실측 확인.
7. **단어 추천 입력** 추가 — 단, 받는 곳이 없다. 배포 전 체크리스트에 올렸다.

**검증**: 랜딩 handoff(질문 중복 없음), Enter 4가지 경우(게이트 잠김/한글 조합
중/Shift+Enter/일반), 키워드 매칭("국민이 주인인 나라" → 주권만 ✓), 4상태 격자,
화면 높이 실측(전부 812px 이내). 빌드/린트 통과.

**아직 안 한 것**: 실제 배포, OG 이미지, 백엔드, 실제 사용자 반응.

### 2026-07-17 — 백엔드 연결 (현재)

VM 한 대에 전부 올리는 구성으로 백엔드를 붙였다. 배포 전 체크리스트의 거짓말
두 개(가짜 통계, 받는 곳 없는 추천 상자)가 이걸로 해소됐다.

**결정 4개** (전부 사용자 확인):

| 질문 | 선택 | 버린 것 |
| --- | --- | --- |
| 어디에 | VM 한 대에 전부 (caddy+web+api+db) | Vercel 분리 — 설정할 게 두 배 |
| HTTPS | 도메인 + Caddy 자동 발급 | — (선택 아님. §아래) |
| 원문 저장 | **안 함.** 추천 단어만 저장 | 원문 데이터 전부 (6.2) |
| DB | PostgreSQL 17 | SQLite |

**HTTPS가 선택이 아닌 이유**: 공유 버튼(Web Share API)과 클립보드 복사가
secure context에서만 동작한다. http로 띄우면 이 프로젝트의 **존재 이유인 공유
기능이 조용히 죽는다.** 그리고 Let's Encrypt는 IP로 인증서를 발급하지 않으므로
도메인이 필수다(없으면 sslip.io).

**가짜 통계를 어떻게 없앴나**: `data/seedStats.ts`와 `STATS_ARE_PLACEHOLDER`를
삭제하고 실제 집계로 바꿨다. 다만 **백엔드를 붙였다고 정직해지는 게 아니었다** —
표본 3명으로 "33%만 설명해냈습니다"를 띄우면 출처만 진짜지 거짓말인 건 똑같다.
그래서 `MIN_SAMPLE=30` 미만이면 서버가 `null`을 주고 UI가 문구를 통째로 숨긴다.
**공개 직후 랜딩에 사회적 증거가 없는 게 정상이고, 그게 정직한 상태다.**

**화이트리스트를 서버에 둔 이유**: "원문은 저장 안 한다"를 주석으로 적으면
언젠가 깨진다. `server/src/events.ts`의 `ALLOWED_PROPS`에 없는 필드는 프론트가
뭘 보내든 버려진다. 결정을 관례가 아니라 구조로 만들었다.

**관리자 대시보드는 안 만들었다.** `GET /api/admin/summary` JSON 하나뿐이고
curl로 본다. 다만 이 엔드포인트 자체는 뺄 수 없다 — 옵션 2가 "반응을 바탕으로
실제로 수정"할 것을 요구하는데 수치를 볼 방법이 없으면 과제를 수행할 수 없다.

**검증한 것**:
- 화이트리스트(`clean()`) 단위 테스트 11건 통과. 가짜 개인정보를 넣은 `text`
  필드가 실제로 버려지는 것, 추천 단어 40자 컷, 모르는 이벤트 거부, UTM 통과 확인
- 서버 타입체크, `next build`(전 라우트 Static/SSG 유지, `/`는 ISR 5분),
  `eslint`, `docker compose config` 전부 통과

⚠️ **검증하지 못한 것 — VM 첫 기동 때 반드시 확인할 것**:
- **집계 SQL** (`server/src/stats.ts`) — 착각률이 맞게 나오는지. 제일 위험한 부분이다
- **이벤트 적재** — `POST /api/events` → Postgres
- **도커 빌드 자체** — Dockerfile 두 개가 실제로 빌드되는지
- **Caddy 라우팅** — `/api/*` 분기와 인증서 발급

이유: 로컬 Docker Desktop이 시작 단계에서 깨져 있었다. `%LOCALAPPDATA%\Docker\run\
dockerInference` 소켓 파일이 고아 상태로 남아 Windows가 삭제조차 거부했고,
Inference manager가 거기 바인딩하려다 엔진 전체가 안 떴다. 우리 코드와 무관하고
재부팅해야 풀린다. `EnableDockerAI: false`로 우회를 시도했으나 **효과 없었다** —
그 플래그는 Inference manager 기동을 막지 않는다.

검증 스크립트는 써 뒀다(§8.2의 curl 절차). Docker가 살아나면 그대로 돌리면 된다.
