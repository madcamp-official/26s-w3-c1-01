## 1. 🔴 파라미터 4개 중 2개는 전달 경로가 없다

**대상: D**

A 명세 §2는 파라미터 4개를 정의하고, "실제 값은 D가 관리하는 `CalibrationProfile`에서
주입받는다. A는 자체 설정 UI를 갖지 않는다"고 명시했다.

| 파라미터 | CalibrationProfile 필드 | 상태 |
| --- | --- | --- |
| 이동 속도(Sensitivity) | `sensitivityLevel` | ✅ 있음 |
| Smoothing 강도 | `smoothingLevel` | ✅ 있음 |
| 화면 구역 이동(N×N) | — | ❌ **없음** |
| 시선 보조 가중치 | — | ❌ **없음** |

즉 A는 뒤의 두 값을 **받을 방법이 없다.** A가 자체 UI를 갖지 않기로 했으므로 사용자가
바꿀 수단도 없다.

**A의 임시 결정**: 두 값을 A 내부 상수로 두고 구현한다
(시선 보조 가중치 = 0.2, 화면 구역 이동 = 비활성). 통합 시 `CalibrationProfile`에
`gazeAssistWeight`, `gridSize` 필드를 추가하면 상수를 걷어내는 것으로 끝난다.

**필요한 결정**: 스키마에 2개 필드를 추가할지, 아니면 MVP에서 두 기능을 잘라낼지.

## 2. ⚪ "화면 구역 이동"은 정의 자체가 없다

**대상: 기획/D**

"화면을 N×N 구역으로 나누어 큰 폭으로 이동(기본 3×3)"이라고만 돼 있다. 그런데 이건
연속 좌표 매핑과 **양립하지 않는 별개의 조작 방식**이다. 문서 어디에도 다음이 없다:

- 연속 매핑과 동시에 동작하는가, 아니면 대체 모드인가
- 대체 모드라면 무엇으로 전환하는가 (음성 명령? B의 사전에는 해당 명령이 없다)
- 구역 이동 시 포인터는 구역 중앙으로 점프하는가

**A의 임시 결정**: **구현하지 않는다.** 5일 일정에서 가장 먼저 잘라낼 후보로 본다.
연속 매핑만으로 모든 시나리오(1~4)가 커버되고, 시나리오 대응 매핑 표에도 구역 이동은
등장하지 않는다.

**필요한 결정**: 정식으로 MVP 범위에서 제외할지 확인.

## 3. 🔴 `x`, `y`가 null이 될 수 있는지 스키마가 서로 모순

**대상: C, D**

| 위치 | 내용 |
| --- | --- |
| A 명세 §3.2 스키마 | `"x": 0.482, "y": 0.317` — 숫자 |
| A 명세 §4 예외 표 | "`CalibrationProfile` 미수신 상태 → 좌표 **null**" |
| C 명세 §3.1 | 항상 숫자 가정. "가장 최근 `PointerFrame`을 현재 좌표로 사용" |

C가 null을 받으면 좌표 계산에서 터진다. C 명세에는 null 분기가 없다.

**A의 임시 결정**: **null을 방출하지 않는다.** 프로파일 미수신 시
`faceDetected: false` + `x: 0.5, y: 0.5`(화면 중앙)로 방출한다.
C/D는 `faceDetected` 하나만 보면 되고 null 분기가 생기지 않는다.

**필요한 결정**: 위 방식으로 A 명세 §4를 수정할지 확인. (C 코드에 null 분기를 넣는
것보다 A가 안 보내는 쪽이 싸다)

## 4. 🟡 `confidence` 하나에 성격이 다른 두 신호가 섞여 있다

**대상: D**

A 명세 §4는 "저조도 환경 → `confidence` 값을 낮춰서 방출 → D가 임계값 이하일 때 안내
문구 표시"라고 한다. 그런데 `confidence`는 §3.2에서 얼굴 검출 신뢰도이기도 하다.

**이 둘은 같은 값이 아니다.** 어두워도 얼굴 검출은 높은 신뢰도로 성공할 수 있고,
밝아도 얼굴이 기울면 신뢰도가 떨어진다. 하나로 합치면 D의 "조명을 밝게 해주세요"
안내가 엉뚱한 상황에 뜬다.

**A의 임시 결정**: `confidence`는 **검출 신뢰도만** 의미하도록 구현한다. 저조도는
별도 필드(`lowLight: boolean`, 프레임 평균 휘도 기준)로 함께 방출한다. D가 안 쓰면
무시하면 되므로 추가해도 깨지지 않는다.

**필요한 결정**: D가 `lowLight`를 쓸지, 아니면 저조도 안내를 MVP에서 뺄지.

## 5. 🔴 전면 카메라 미러링과 화면 회전 규약이 어디에도 없다

**대상: C, D (좌표를 소비하는 쪽 전부)**

문서 4종 어디에도 다음이 정의돼 있지 않다. 통합 시 "포인터가 반대로 움직인다"로
반드시 나타난다.

- **미러링**: 전면 카메라 영상은 좌우가 뒤집혀 있다. 사용자가 고개를 오른쪽으로 돌리면
  영상에서는 왼쪽이다. 포인터는 어느 쪽으로 가야 하는가
- **yaw/pitch 부호**: 오른쪽이 +인가 -인가. 위가 +인가 -인가
- **화면 회전**: 시나리오 4는 태블릿 가로 모드다. `x`,`y` 0~1은 **세로 기준 화면**인가
  **현재 회전 상태의 화면**인가
- **0~1의 기준**: 전체 화면인가, 상태바/네비바를 제외한 영역인가

**A의 임시 결정**:
- 사용자가 고개를 **오른쪽으로 돌리면 포인터도 오른쪽(x 증가)** — 거울이 아니라
  사용자 기준. 직관적으로 이쪽이 맞다.
- `yaw` +가 오른쪽, `pitch` +가 위.
- `x`,`y`는 **현재 회전 상태의 전체 화면** 기준 0~1. 회전은 A가 흡수해서 소비 측은
  회전을 신경 쓰지 않아도 되게 한다.

**필요한 결정**: C가 OS 이벤트를 주입할 때 쓰는 좌표계와 위 기준이 일치하는지 확인.
**여기가 어긋나면 터치가 엉뚱한 곳에 찍힌다.**

## 6. 🟡 `referencePoints[9]`를 A가 사용하지 않는다

**대상: D**

D는 캘리브레이션에서 9개 기준점을 수집해 `referencePoints[9]`로 넘긴다. 그런데 A의
처리 로직(§2-3)은 "`CalibrationProfile` 기준으로 정규화"라고만 돼 있고, 스키마에
`faceRangeYaw/Pitch` min/max가 별도로 있다.

**A가 min/max만 쓰면 선형 정규화가 되고, `referencePoints[9]`는 쓰이지 않는다.**
9점을 쓰려면 보간(bilinear/homography)을 해야 하는데 이건 정확도는 올라가지만
구현·튜닝 비용이 크다.

**A의 임시 결정**: **선형 정규화만 구현한다**(min/max 사용). MVP에서 9점 보간은
과하다고 판단했다. `referencePoints`는 받아서 min/max 산출에만 쓰고 보관한다.

**필요한 결정**: D가 "9점 보간은 A가 한다"고 기대하고 있었는지 확인.
**기대가 어긋나면 캘리브레이션 9단계를 사용자에게 시키고도 정확도가 안 나온다** —
그럴 바엔 D가 기준점을 5개(중앙+상하좌우)로 줄이는 게 사용자 시간을 아낀다.

## 7. 🟡 타임스탬프 기준 시계가 정의돼 있지 않다

**대상: B, C, D**

A(`PointerFrame`), B(`VoiceCommandEvent`), C(`ExecutionResult`) 모두 `timestamp`를
방출하고 D가 이걸로 순서를 판단한다. 그런데 기준 시계가 어디에도 없다.
벽시계(`System.currentTimeMillis`)는 시간 동기화로 튈 수 있고, 단조 시계
(`SystemClock.elapsedRealtime`)는 안 튄다.

**A의 임시 결정**: 단조 시계(`elapsedRealtime`) 기준 밀리초.

**필요한 결정**: B/C도 같은 시계를 쓰는지. 섞이면 D의 순서 판단이 조용히 틀어진다.

## 8. ⚪ 에러 이벤트가 인터페이스 계약에 없다

**대상: D**

A 명세 §4는 "카메라 권한 거부 → `CameraPermissionError`를 D에 전달"이라고 하는데,
§3 인터페이스 계약의 출력 목록에는 `RawFaceOrientation`과 `PointerFrame`뿐이다.
`CameraPermissionError`의 스키마가 정의된 곳이 없다(B의 `MicPermissionError`도 동일).

**A의 임시 결정**: `{ "type": "CAMERA_PERMISSION_DENIED", "timestamp": ... }` 형태의
단순 이벤트로 방출한다. B와 형태를 맞추는 게 좋으므로 B 담당자와 짧게 맞출 것.
