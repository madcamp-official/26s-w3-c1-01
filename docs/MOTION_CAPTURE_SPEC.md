## 핸즈프리 스마트폰 컨트롤러 · 신규 입력 모달리티: 손 동작(모션 캡쳐)

> 기반 문서: [FUNCTIONAL_SPEC.md](FUNCTIONAL_SPEC.md) · [INTEGRATION.md](INTEGRATION.md) · [PART_D_SPEC.md](PART_D_SPEC.md)
> **상태: 초안. 팀 합의 및 실기기 검증 전.** 아래 §9(미결정/리스크)를 먼저 읽을 것.

## 1. 문서 개요

### 1.1 목적

현재 컨트롤러는 **포인터 = 얼굴/시선(A)**, **트리거 = 음성(B)** 두 채널로 동작한다.
여기에 **전면 카메라로 손 동작(모션 캡쳐)을 추적**해, 손으로 포인터를 움직이고
손 제스처로 명령을 내리는 **세 번째 입력 모달리티**를 추가한다.

**핵심 설계 원칙: D의 포트 4개를 건드리지 않는다.** 모션 캡쳐는 기존
`PointerSource`/명령 스트림에 꽂히는 **새 어댑터**로만 구현하고, D의 상태 머신
(`CommandGate`/`StateHolder`/`TransitionTable`), C의 실행부(`InputExecutionEngine`),
오버레이(`OverlayView`)는 **한 줄도 바꾸지 않고 재사용**한다. 이게 이 기능을
저비용으로 넣을 수 있는 이유이자, 이 명세의 유일한 절대 조건이다.

### 1.2 범위

**포함(이번 단계)**
- 손 랜드마크 기반 포인터 이동(검지 끝 좌표 → 화면 좌표)
- 정적/동적 손 제스처 → 기존 `CommandId`(17종)로 매핑
- 입력 모드 전환(FACE / HAND / HYBRID)과 모드별 캘리브레이션
- 카메라 프레임 1스트림을 얼굴·손 랜드마커에 분배(fan-out)

**제외(다음 단계)**
- 전신 포즈(Pose Landmarker) 기반 제어 — §8에 확장 훅만 남긴다
- 양손 동시 추적, 손 기반 다중 사용자 식별
- 커스텀 제스처 등록, 제스처 학습(온디바이스 파인튜닝)
- 원격/외부 카메라, 3D 깊이 센서

### 1.3 용어

| 용어 | 정의 |
| --- | --- |
| 손 포인터 | 손 랜드마크(기본: 검지 끝, landmark #8)를 화면 좌표에 매핑한 커서 |
| 정적 제스처 | 한 프레임의 손 모양으로 판정하는 명령(주먹, 편 손바닥, 핀치 등) |
| 동적 제스처 | 시간에 따른 손 궤적으로 판정하는 명령(스와이프 상/하/좌/우) |
| 입력 모드 | 활성 입력원 조합. FACE / HAND / HYBRID 중 하나 |
| 제스처 게이트 | 오검출 억제 장치. 최소 유지 프레임·쿨다운·신뢰도 임계값 |
| Dwell(체류) | FUNCTIONAL_SPEC과 동일. **자동 실행 트리거로는 여전히 미사용** |

## 2. 기존 아키텍처와의 접점

### 2.1 어디에 꽂히는가

```
[전면 카메라 1스트림 (CameraX ImageAnalysis)]
        │  프레임 fan-out (신규: FrameHub)
        ├──────────────► FaceLandmarker  → (A) FaceTracker      → PointerFrame  ┐
        └──────────────► HandLandmarker  → (신규) HandTracker    → PointerFrame  ┤→ PointerSource (D포트, 무변경)
                                                    └ 제스처 판정 → CommandId ─────┐
[마이크] → (B) VoiceRecognizer → VoiceCommandEvent ────────────────────────────┤→ 명령 스트림 → D CommandGate (무변경)
                                                                                 ┘
                D CommandGate(무변경) → ExecutionSink(C, 무변경) → OS 이벤트
```

포인터와 명령은 FUNCTIONAL_SPEC §2.2와 동일하게 **병렬**로 흐르고, **명령 실행 시점의
포인터 좌표** 기준으로 동작한다. 모션 캡쳐는 이 두 흐름에 **입력원을 하나씩 더** 붙이는 것뿐이다.

### 2.2 재사용하는 기존 계약(변경 없음)

| 타입/포트 | 위치 | 모션 캡쳐에서의 역할 |
| --- | --- | --- |
| `PointerFrame` | `core.model` | 손 포인터도 **같은 타입**으로 방출. x/y 정규화 0~1, `faceDetected`는 `handDetected` 의미로 재사용(§9-A 참고) |
| `PointerSource` | `orchestrator.port` | `HandPointerSource`가 이 포트를 구현. D는 얼굴/손을 구분하지 않는다 |
| `CommandId` | `core.model` | 제스처 → 이 enum 17종으로 정규화. **새 값 추가하지 않는다** |
| `VoiceCommandEvent` | `core.model` | 제스처 명령을 이 이벤트로 감싸 명령 스트림에 합류(§4.3) |
| `CommandGate`/`StateHolder` | `orchestrator.state` | 상태×명령 게이트키핑 그대로. 85조합 유닛테스트 무변경 |
| `ExecutionSink` | `orchestrator.port` (C 구현) | 탭/드래그/스크롤 실제 주입. 무변경 |
| `CalibrationConsumer` | `orchestrator.port` | 손 도달 범위 캘리브레이션 프로파일 전달에 재사용(§6) |

### 2.3 신규 코드(예정)

기존 `com.madcamp.handsfree.integration`이 A/B/C를 D 포트로 이어붙인 것과 **같은 패턴**을 따른다.

| 파일(안) | 패키지 | 하는 일 |
| --- | --- | --- |
| `HandTracker` | `com.madcamp.handsfree.tracking` | HandLandmarker → `PointerFrame` + 제스처 이벤트. A의 `FaceTracker`와 대칭 |
| `HandLandmarkMapper` | `.tracking` | 손 랜드마크 → 정규화 좌표. A의 `PointerMapper` 대응 |
| `GestureClassifier` | `.tracking` | 랜드마크 → 정적/동적 제스처 판정(§4) |
| `HandPointerSource` | `.integration` | `HandTracker` → D `PointerSource`. A의 `FacePointerSource`와 대칭 |
| `GestureCommandSource` | `.integration` | 제스처 → `CommandId` → `VoiceCommandEvent` 어댑터(§4.3) |
| `FrameHub` | `.tracking` | 단일 카메라 프레임을 얼굴/손 랜드마커에 분배(§7) |
| `InputModeController` | `.integration` | FACE/HAND/HYBRID 전환과 입력원 활성/비활성 |

## 3. 기능 명세 — 손 포인터

### FR-M-001 손 포인터 이동

**입력**: 전면 카메라 프레임 스트림(권장 15fps 이상), 손 캘리브레이션 프로파일

**처리**: HandLandmarker로 손 랜드마크 21점 검출 → 기준점(기본 검지 끝 #8, 옵션 손목 #0)
좌표 추출 → 손 도달 범위 프로파일로 정규화 → 화면 좌표 매핑 → Smoothing 필터(A의
`Smoother` 재사용 가능) → 감도 스케일링 → `PointerFrame` 방출.

**세부 파라미터**

| 파라미터 | 설명 | 기본값 | 조정 범위 |
| --- | --- | --- | --- |
| 추적 기준점 | 포인터로 삼을 랜드마크 | 검지 끝(#8) | 검지끝 / 손목 / 손 중심 |
| 이동 속도(Sensitivity) | 손 이동 대비 포인터 이동량 | 중간 | 낮음/중간/높음 |
| Smoothing 강도 | 떨림 저감 | 중간 | 낮음/중간/높음 |
| 데드존 | 미세 떨림 무시 반경(정규화) | 0.01 | 0~0.05 |

**좌표 규약(A와 동일하게 맞춘다 — 어긋나면 실기기에서만 드러난다)**
- 미러링: **손을 사용자 기준 오른쪽으로 → x 증가**(전면 카메라 좌우 반전 흡수).
  A의 `PointerFrame` 규약과 반드시 일치시킨다(INTEGRATION §5.2-1과 같은 종류의 위험).
- 기기 회전 보정은 트래커가 흡수. C/D는 신경 쓰지 않는다.
- 손 미검출 시에도 `PointerFrame`은 **null이 되지 않는다** — 마지막 좌표 유지 +
  `handDetected=false`.

**예외**

| 상황 | 처리 |
| --- | --- |
| 손 미검출 | 포인터를 **마지막 위치에 고정**, "손 인식 안됨" 안내 |
| 프레임에 손이 여럿 | **가장 큰/가장 앞선 손 1개만** 사용(양손은 다음 단계) |
| 도달 범위 초과 | 화면 경계값으로 **클램핑**(FR-001과 동일) |
| 손이 화면 밖으로 이탈 | 3초 유예 후 미검출로 처리 |
| 카메라 권한 거부 | FR-001과 동일 처리 |

## 4. 기능 명세 — 손 제스처 명령

> 제스처는 기존 `CommandId` 17종으로만 매핑한다. **enum에 값을 추가하지 않는다.**
> 매핑되지 않는 제스처는 존재하지 않는 것으로 취급한다.

### FR-M-002 정적 제스처 (손 모양)

| 제스처 | 판정(요지) | 매핑 `CommandId` | 유효 상태 |
| --- | --- | --- | --- |
| 핀치(엄지+검지 붙음) | #4·#8 거리 임계값 이하 | `TOUCH` | ACTIVE |
| 주먹 쥐기 | 네 손가락 모두 접힘 | `DRAG_START` | ACTIVE |
| 손 펴기(주먹 해제) | 네 손가락 모두 폄 | `DRAG_END` | DRAGGING |
| 편 손바닥 정지(hold 1s) | 손바닥 정면 + 정지 유지 | `STOP` | ACTIVE, DRAGGING |
| 엄지 척(정지 상태에서) | thumbs-up 형태 | `RESUME` | PAUSED |
| ✊ 유지(hold 2s) | 주먹 장시간 유지 | `LOCK` | ACTIVE, PAUSED |
| 🖐 유지(hold 2s, LOCKED) | 편 손 장시간 유지 | `UNLOCK` | LOCKED |

> ⚠️ **주먹의 이중 매핑(DRAG_START vs LOCK)**은 유지 시간으로만 구분한다. 짧게=드래그,
> 길게=잠금. 오작동 위험이 커서 §9-C의 팀 결정 대상이다. 대안: 잠금은 손 제스처에서
> 빼고 화면 버튼/음성 전용으로 둔다(FR-005의 수동 해제 버튼은 어차피 상시 노출).

### FR-M-003 동적 제스처 (스와이프)

방향과 속도/거리로 강도를 함께 파싱한다(FR-004의 강도 체계 재사용).

| 제스처 | 매핑 `CommandId` | 비고 |
| --- | --- | --- |
| 위로 스와이프 | `SCROLL_UP` | 짧게=`_SMALL`, 크게=`_LARGE` |
| 아래로 스와이프 | `SCROLL_DOWN` | 위와 동일 |
| 왼쪽 스와이프 | `PREV` | 발표 시나리오(좌 스와이프 이벤트) |
| 오른쪽 스와이프 | `NEXT` | 발표 시나리오 |
| 편 손 좌→우 크게 휘두름 | `BACK` | 뒤로가기(취소). §9-C 확정 필요 |

### FR-M-004 제스처 게이트(오검출 억제)

음성의 신뢰도 임계값(B, 0.6)에 대응하는 안전장치. **제스처는 오검출이 잦아 이 게이트가 없으면 못 쓴다.**

| 파라미터 | 설명 | 기본값 |
| --- | --- | --- |
| 최소 유지 프레임 | 정적 제스처가 명령이 되기 위한 연속 프레임 수 | 5프레임(~0.3s @15fps) |
| 쿨다운 | 같은 명령 재발화 최소 간격 | 800ms |
| 랜드마크 신뢰도 임계값 | HandLandmarker 점수 하한 | 0.6 |
| 스와이프 최소 이동 | 동적 제스처 인정 이동량(정규화) | 0.15 |
| 스와이프 최대 시간 | 이 시간 초과 궤적은 스와이프 아님 | 500ms |

**예외**

| 상황 | 처리 |
| --- | --- |
| 신뢰도 임계값 미달 | 명령 무시(발화 없음). 음성 소음 처리와 동형 |
| 쿨다운 내 재검출 | 무시 |
| 포인터 이동과 제스처 동시 발생 | 제스처 판정 우선, 판정 프레임 동안 포인터 홀드 |
| 애매한 손 모양 | 명령 없음(오탐보다 미탐을 택함) |

### 4.3 제스처 → 명령 스트림 합류

`GestureCommandSource`는 판정된 제스처를 **`VoiceCommandEvent`로 감싸** 명령 스트림에 넣는다.
D의 `CommandGate`는 명령의 **출처(음성/제스처)를 구분하지 않는다** — `CommandId`만 본다.

- 구현 선택지 (§9-B 결정 대상):
  1. `GestureCommandSource`가 `VoiceCommandSource`를 그대로 구현(events: Flow<VoiceCommandEvent>) →
     `RealConductorDependencies`에서 **음성 Flow와 `merge`**해 하나의 `voiceCommandSource`로 넘긴다. **포트 무변경.** (권장)
  2. D에 명령원 포트를 하나 더 두고 오케스트레이터가 두 스트림을 구독 → 포트 변경 발생. **비권장.**
- `VoiceCommandEvent`에 출처 태그(`source: VOICE|GESTURE`)를 **선택적 필드로만** 추가하면
  오버레이 피드백 구분이 가능하되, 기본값을 둬 기존 생성부를 깨지 않는다(INTEGRATION §2.3의
  "필드는 뒤에 기본값으로만 추가" 규칙 준수).

## 5. 입력 모드

### FR-M-005 모드 전환

| 모드 | 포인터 입력원 | 명령 입력원 | 용도 |
| --- | --- | --- | --- |
| FACE (기본, 현행) | 얼굴/시선(A) | 음성(B) | 손 사용 제한 사용자. 기존 동작 그대로 |
| HAND | 손(신규) | 손 제스처 + 음성 | 조용해야 하는 환경, 발표, 음성 인식 불가 상황 |
| HYBRID | 얼굴 **또는** 손(우세한 쪽) | 음성 + 제스처 모두 | 상황 적응. 카메라 부하 최대(§7) |

- 전환은 설정 화면 및 음성 명령(예: "손 모드")으로 가능. **전환 시 진행 중 드래그는 안전하게 취소.**
- **접근성 최우선 원칙 유지**: FACE 모드와 음성 안전 명령("멈춰"/"잠금")은 어느 모드에서도
  살아 있어야 한다(NFR). 손이 안 보여도 음성으로 잠글 수 있어야 한다.

### 5.1 HYBRID 입력원 중재

두 포인터원이 동시에 값을 낼 때 규칙:
- 손이 검출되면 **손 우선**(의도적 조작으로 간주), 손 미검출 시 얼굴로 폴백.
- 전환 시 포인터 점프 방지를 위해 전환 프레임에서 **좌표 보간(램프)** 적용.
- 두 명령원은 그대로 병합(§4.3). 안전 명령 충돌은 FUNCTIONAL_SPEC §8과 동일하게 **안전 우선**.

## 6. 캘리브레이션 (손)

FR-006(9점 얼굴 캘리브레이션)과 **대칭 구조**로, 손 도달 범위를 보정한다.

| 단계 | 내용 |
| --- | --- |
| 1 | 기기 거치 및 손을 카메라 화각 안에 두도록 안내 |
| 2 | 화면 중앙을 가리키도록(중앙 기준점) |
| 3 | 상/하/좌/우 + 대각선 총 **9개 기준점**을 손으로 순차 지시 |
| 4 | 손 이동 범위(x/y min/max) 저장, 진행률 표시 |
| 5 | 테스트 포인터로 감도 조정 + 핀치 임계값 개인화 |

**프로파일 스키마(안, 손 전용 확장)**: `profile_id` · `hand_reach_x_min/max` ·
`hand_reach_y_min/max` · `pinch_threshold` · `handedness`(좌/우손) · `sensitivity_level` ·
`smoothing_level` · `created_at`/`updated_at` — **기기 로컬 저장**.

- 기존 `CalibrationProfile`(`core.model`)에 손 필드를 **기본값과 함께 뒤에 추가**하거나,
  별도 `HandCalibrationProfile`로 분리한다(§9-D). 어느 쪽이든 A의 얼굴 프로파일과 충돌 없어야 한다.
- 재보정/중도이탈/반복실패 예외 처리는 FR-006과 동일.

## 7. 카메라 자원 — 단일 스트림 fan-out (가장 중요한 통합 리스크)

**전면 카메라는 하나다.** FaceTracker와 HandTracker가 **각자 CameraX를 열 수 없다** — 두
번째 `bindToLifecycle`은 충돌하거나 프레임을 굶긴다.

**해결(FrameHub)**: CameraX `ImageAnalysis`를 **한 번만** 바인딩하고, 각 프레임을
활성화된 랜드마커(들)에 분배한다.
- FACE 모드: 얼굴 랜드마커만 구동.
- HAND 모드: 손 랜드마커만 구동.
- HYBRID: 한 프레임을 두 랜드마커에 전달(부하 2배). 필요 시 **프레임 스킵**(예: 손은
  매 프레임, 얼굴은 격 프레임)으로 fps 방어.
- 두 랜드마커 모두 MediaPipe `tasks-vision:0.10.18`이 이미 제공(모델 파일만 추가).
- 현재 `FaceTracker`가 카메라를 직접 소유하므로(카메라 바인딩이 트래커 안에 있음),
  **FrameHub 도입은 A의 `FaceTracker`에서 카메라 바인딩부를 분리하는 리팩터링을 수반**한다.
  이건 A파트 코드 변경이므로 §9-E에서 A와 합의가 필요하다.

**필요 에셋**: `app/src/main/assets/hand_landmarker.task`(약 7~8MB).
`face_landmarker.task`처럼 **gitignore 대상일 가능성**이 높다 — 없으면 실행 즉시
`MODEL_LOAD_FAILED`. INTEGRATION §5.1-3과 동일한 함정이므로 설치 가이드에 명시한다.

## 8. 확장 훅 — 전신 포즈 (다음 단계, 비구현)

Pose Landmarker를 붙이면 팔/어깨 큰 동작으로 원거리 제어(예: 발표 중 몇 미터 밖에서)가 가능하다.
- 같은 `PointerSource`/명령 스트림 구조를 그대로 재사용 → 새 어댑터만 추가.
- 손 대신 손목/팔꿈치 랜드마크를 포인터 기준점으로. 제스처는 팔 스윙으로 확장.
- 이번 단계에서는 **인터페이스만 열어두고 구현하지 않는다.**

## 10. 비기능 요구사항 (기존 NFR에 추가)

| 구분 | 요구사항 |
| --- | --- |
| 지연시간 | 손 인식~포인터 반영 **200ms 이내**(FR-001과 동일 목표) |
| 프레임 | 활성 모드에서 **최소 15fps**. HYBRID는 프레임 스킵 허용 |
| 자원 | 카메라 1스트림 원칙 준수(§7). 두 번째 CameraX 바인딩 금지 |
| 개인정보보호 | **온디바이스 처리, 원본 영상 저장 안 함**(캘리브레이션 수치만 로컬) — 기존 원칙 그대로 |
| 안전장치 | **"멈춰"/"잠금"은 손 모드에서도 음성으로 항상 최우선 처리**. 손이 안 보여도 잠글 수 있어야 한다 |
| 하위호환 | FACE 모드 = 현행 동작. 계약 타입에 필드 추가는 **뒤에 기본값으로만**(INTEGRATION §2.3 규약) |

## 11. 검증 순서 (구현 후, INTEGRATION §5 형식)

1. **좌표 미러링** — 손 x축이 A의 얼굴 규약과 일치하는가. 어긋나면 화면 반대편에 터치.
2. `hand_landmarker.task` 에셋 존재 확인(없으면 `MODEL_LOAD_FAILED`).
3. **단일 카메라 fan-out** — FrameHub가 두 랜드마커에 프레임을 굶기지 않고 분배하는가.
4. **제스처 오검출률** — 게이트(§4.4) 기본값으로 실사용 가능한지. 오탐 vs 미탐 균형.
5. **모드 전환 중 드래그 안전 취소**, HYBRID 포인터 점프 없음.
6. **성능** — HAND 단독, HYBRID+음성 각각의 fps/발열 실측(§9-F).
7. **접근성 회귀** — FACE+음성 경로가 이 변경으로 전혀 영향받지 않는지(§9-G).
