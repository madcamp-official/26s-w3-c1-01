# 기능 명세서 [B] — 음성 인식 및 명령 해석 엔진

- 담당 FR: FR-007 (음성 명령 인식 및 해석)
- 역할 한 줄 요약: 마이크를 듣고 "정규화된 명령 ID"를 뱉어내는 센서 모듈. A와 마찬가지로 컨트롤러 상태를 몰라도 된다 — 명령이 지금 유효한지 판단은 전부 D가 한다.

---

## 1. 역할 개요 (Scope)

**한다**

- 마이크 입력 실시간 음성 인식(STT)
- 인식된 텍스트를 사전 정의된 명령어 사전과 매칭 (동의어 포함)
- 시스템 전체에서 쓰이는 **모든** 음성 명령의 사전을 단독으로 소유 (터치/드래그/스크롤/제어 명령 전부 포함)
- 인식 신뢰도가 낮거나 사전에 없는 발화는 무시 처리

**하지 않는다 (다른 역할 책임)**

- 지금 이 명령이 실행 가능한 상태인지 판단 → D 책임 (예: PAUSED 상태에서 "터치"가 들어와도 B는 그냥 `TOUCH` 이벤트를 D에 전달할 뿐, 실행 여부는 D가 결정)
- 실제 탭/드래그/스크롤 실행 → C 책임
- 포인터 위치 계산 → A 책임

> B는 "언어를 이해하는 번역기"에만 집중한다. "터치", "클릭"처럼 같은 의미의 다른 표현을 하나의 `commandId`로 정규화해서 넘기면, C와 D는 한국어 동의어를 전혀 몰라도 되도록 만드는 것이 핵심 설계 의도다.
>

---

## 2. 명령어 사전 (B가 단독 소유)

| commandId | 인식 발화(동의어 포함) | 비고 |
| --- | --- | --- |
| TOUCH | 터치, 클릭 |  |
| BACK | 취소 | 뒤로가기 |
| DRAG_START | 잡아, 드래그 시작 |  |
| DRAG_END | 놓아 |  |
| DRAG_CANCEL | 드래그 취소 |  |
| SCROLL_DOWN | 아래로 |  |
| SCROLL_UP | 위로 |  |
| SCROLL_DOWN_SMALL | 조금 아래로 |  |
| SCROLL_UP_SMALL | 조금 위로 |  |
| SCROLL_DOWN_LARGE | 크게 아래로 |  |
| SCROLL_UP_LARGE | 크게 위로 |  |
| STOP | 멈춰 |  |
| RESUME | 다시 시작 |  |
| LOCK | 잠금 |  |
| UNLOCK | 해제 |  |
| NEXT | 다음 | 발표 시나리오용 |
| PREV | 이전 | 발표 시나리오용 |

> 이 표가 시스템 전체의 "단일 진실 공급원(SSOT)"이다. C, D는 이 `commandId` 값만 참조하고, 실제 한국어 문자열은 전혀 참조하지 않는다. 명령어를 추가/변경할 때는 이 표만 수정하면 된다(Phase 2에서 사용자 커스텀 명령어 등록 시에도 이 사전 구조를 확장).
>

---

## 3. 처리 로직

1. 마이크 입력을 STT 엔진에 전달
2. 인식된 텍스트를 명령어 사전과 매칭 (동의어 정규화 포함)
3. 매칭 성공 시 `VoiceCommandEvent` 생성 후 D로 전달
4. 매칭 실패 또는 신뢰도 임계값 미달 시 이벤트를 발생시키지 않음 (선택적으로 `UnrecognizedSpeech` 디버그 이벤트만 발생)

---

## 4. 인터페이스 계약

### 4.1 출력 `VoiceCommandEvent` (B → D 단일 수신자)

B는 오직 D에게만 전달한다. C에게 직접 보내지 않는다 (C는 D가 유효성 검증 후 내려주는 `ExecutionCommand`만 받는다).

```json
{
  "commandId": "TOUCH",
  "rawText": "터치",
  "confidence": 0.94,
  "timestamp": 1721270000000
}
```

### 4.2 입력 없음

B는 다른 모듈의 데이터를 구독하지 않는다 (완전 독립 모듈). 단, Phase 2에서 화자 인식 필터링 도입 시 등록된 사용자 음성 프로파일을 입력받을 수 있음(본 MVP 범위 외).

---

## 5. 예외 처리

| 상황 | 처리 방안 |
| --- | --- |
| 인식된 텍스트가 사전에 없음 | 이벤트 미발생, 필요 시 `UnrecognizedSpeech` 디버그 로그만 기록 |
| 소음이 많은 환경 (신뢰도 낮음) | `confidence` 임계값(예: 0.6) 미달 시 이벤트 미발생 |
| 주변 사람 발화로 인한 오작동 | MVP는 임계값 기반 필터링만 적용, 화자 인식은 Phase 2 |
| 마이크 권한 거부 | `VoiceCommandEvent` 대신 `MicPermissionError`를 D에 전달 |
| 연속 발화("아래로 아래로") | 개별 이벤트 2건으로 순차 방출 (합치지 않음) |

## 6. 병렬 개발 전략 (Mock 기준)

- B팀은 D의 상태 머신 존재를 가정하지 않는다. 콘솔에 `VoiceCommandEvent` JSON을 출력하는 것으로 1차 완료 기준을 삼는다.
- 테스트용 발화 샘플 세트(사전의 모든 commandId × 동의어 조합)를 자체적으로 녹음/텍스트로 준비해 매칭 정확도를 검증한다.

## 7. 완료 기준 (Definition of Done)

- [ ]  사전의 모든 명령(17개 commandId, 동의어 포함)이 정확히 매칭됨
- [ ]  신뢰도 임계값 미달 발화는 이벤트가 발생하지 않음
- [ ]  사전에 없는 발화는 무시됨 (오작동 없음)
- [ ]  연속 발화가 개별 이벤트로 분리되어 방출됨
- [ ]  마이크 권한 거부 시 에러 이벤트 정상 전달

## 8. 통합 시 D팀과 확인할 사항

- `confidence` 임계값 최종 수치 합의
- commandId 목록에 추가/변경이 필요할 경우(예: 발표 시나리오의 NEXT/PREV 확정 여부) B의 사전과 D/C의 처리 로직을 동시에 갱신해야 하므로 사전 변경 시 3팀(B,C,D) 공지 필요
